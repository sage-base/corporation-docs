# 直近1年の活動計画（2026年4月〜2027年3月）

## 概要

本ドキュメントは、sagebase プロジェクトにおける 2026年度（2026年4月〜2027年3月）の活動計画をまとめたものである。これまで蓄積してきた国会データを基盤に、**データの意味化・拡充・提供経路の多様化**という3軸で価値提供を強化する。

### これまでの到達点（2026年3月時点）

- 国会会議録API連携・11.2M件の発言データ蓄積（Goal 0004, 0011）
- 発言者-政治家の全量紐付け完了（Goal 0014）
- BigQuery パブリックデータセット公開（Goal 0006）
- データレイヤー再定義・Data Vault 構築完了（Goal 0018）
- 参議院議案賛否データ取得完了（Goal 0023）
- レコード単位データプロベナンス付与完了（Goal 0021）
- BigQuery データ品質モニタリング稼働中（Goal 0022）

### 2026年度の4テーマ

| # | テーマ | ゴール |
|---|-------|--------|
| 1 | 発言者-政治家紐付けカバレッジ向上 | 国会100%、地方議会で拡張可能な基盤整備 |
| 2 | 議案賛否データの全面整備 | 衆参両院＋地方議会での賛否公開 |
| 3 | 地方議会データ拡充（都道府県→市区町村） | 47都道府県議会＋政令指定都市の整備 |
| 4 | データ提供方法の拡充 | Snowflake / MCP / API など多経路での配信 |

## テーマ1: 発言者-政治家紐付けカバレッジ向上

### 背景と課題

Goal 0014 により国会の発言データに対する一次紐付けは完了したが、以下の課題が残る。

- **紐付け率の可視化が不十分**: 発言単位・政治家単位・会議体単位のカバレッジ指標が未整備
- **地方議会対応の未検証**: 国会スキーマで組んだマッチングロジックが地方議員にそのまま使えるかは未検証
- **政治家マスタの網羅性**: 地方議員のマスタデータが整備されないと、地方議会の発言は紐付けできない
- **精度改善の継続**: 同姓同名や改名・所属変更など、エッジケースの追加修正が必要

### 方針

1. **カバレッジ計測の標準化**: 国会・地方議会それぞれで「紐付け済み発言 / 総発言」の集計を dbt test として定義
2. **マッチング基盤の汎用化**: 候補抽出ロジックを会議体種別に依存しない形にリファクタ
3. **地方議員マスタの逐次構築**: 都道府県議会の展開と並行して地方議員データを投入
4. **継続的な精度改善**: ユーザーフィードバック（Goal 0020）経由の訂正を運用に組み込む

### 年度末目標

- 国会発言の紐付けカバレッジ 99% 以上
- 都道府県議会（対応済み自治体）で紐付けカバレッジ 95% 以上
- 紐付けカバレッジダッシュボードを外部公開

## テーマ2: 議案に対する賛否データの作成

### 背景と課題

参議院の個人投票データは Goal 0023 で取得済みだが、以下が未整備。

- **衆議院議案賛否**: 衆議院は公式サイトで個人投票データの構造化が限定的で、追加のスクレイピング・LLM抽出が必要
- **地方議会の賛否データ**: NTCIR PoliInfo で東京都議会の構造化データのみ存在。他自治体はほぼ未整備
- **造反検出**: 会派方針と個人投票の差分を検出する分析レイヤーが未実装
- **議案-投票の紐付け**: 議案マスタと投票レコードの完全一致は 86% 程度で、残り14%はマッチング改善が必要

### 方針

1. **衆議院議案賛否の体系的取得**: 議院運営委員会議事録・本会議議事録からの抽出パイプライン構築
2. **議案-投票マッチング精度の向上**: 議案名正規化とLLMによるあいまいマッチの強化
3. **造反検出レイヤー**: 会派方針推定と個人投票の差分抽出ロジックの実装
4. **地方議会賛否取得**: NTCIR データ投入と、新規自治体の公開データ収集パイプライン整備

### 年度末目標

- 衆議院賛否データ 2020年以降 100% 取得
- 参議院賛否データと議案の紐付けカバレッジ 95% 以上
- 造反検出機能のアルファ版公開（国会のみ）
- 地方議会賛否データの対象自治体 5 以上

## テーマ3: 都道府県議会→市区町村議会の順次データ拡充

### 背景と課題

Goal 0012（京都市議会）でスキーマ汎用性の検証を進行中。ただし、日本の地方議会は以下の構造。

- 47都道府県議会 / 20政令指定都市議会 / 約1,700市区町村議会
- 統一APIは存在せず、kaigiroku.net（DiscussNet系、約540自治体）が最も構造的
- 独自システム・PDF配信・ページ非公開など、自治体ごとに状況が大きく異なる

スケール戦略として、**都道府県議会 → 政令指定都市 → 中核市 → その他**という段階的展開を行う。

### 方針

1. **京都市議会の完全取得**: Goal 0012 の G-1〜G-6 ギャップ解消を完了させ、地方議会向けスキーマ確定
2. **都道府県議会の逐次展開**: kaigiroku.net 採用の都道府県から順に発言・会議体・議員データを取得
3. **政令指定都市の段階展開**: 京都市での知見を活かし、横浜・大阪・名古屋など大都市へ展開
4. **地方議員マスタの構築**: 議会公式サイト・選挙結果データからの逐次投入
5. **収集困難な自治体の特定**: 展開のたびに収集不可理由を記録し、代替手段の検討材料にする

### 年度末目標

- 京都市議会: スキーマギャップ解消完了・データ完全取得
- 都道府県議会: 47都道府県中 30 自治体の発言データ取得
- 政令指定都市: 20市中 10市の発言データ取得
- 地方議員マスタ: 取得済み自治体の現職議員 100%

## テーマ4: データ提供方法の拡充（Snowflake / MCP など）

### 背景と課題

現在の公開経路は **BigQuery パブリックデータセット** のみ。以下の課題がある。

- **Snowflake ユーザーへのリーチ不足**: 企業ユーザーには Snowflake Marketplace の方が扱いやすい
- **AI エージェントからの利用導線がない**: LLM ベースのアプリから sagebase データを直接参照する手段がない
- **API 経由での軽量アクセス未提供**: BigQuery は大規模分析向けで、単発クエリには重い
- **データカタログの不足**: どのテーブルに何があるか、外部ユーザーが把握しにくい

### 方針

1. **Snowflake Marketplace 公開**: BigQuery との同期パイプラインを構築し、主要データセットを公開
2. **MCP サーバー提供**: sagebase データを Claude などの AI エージェントから参照できる MCP サーバーを公開
3. **Popolo 互換 API**: 国際標準の Popolo JSON スキーマでの軽量 REST API 公開（Vision 0006 の具体化）
4. **データインベントリダッシュボード**: Goal 0024 を軸に、データ蓄積状況を可視化する静的サイトを公開
5. **利用事例の蓄積**: 公開したチャネルごとの利用事例を収集し、優先順位付けにフィードバック

### 年度末目標

- Snowflake Marketplace で sagebase 主要データセット公開
- MCP サーバー v1.0 リリース（政治家検索・発言検索・議案検索の3ツール以上）
- Popolo 互換 API アルファ版公開
- データインベントリダッシュボード常時公開

## 依存関係と前提

- **Goal 0018（Data Vault）** が各テーマの基盤として機能する
- **Goal 0020（ユーザーフィードバックループ）** によるデータ訂正が継続される
- **Goal 0022（BQデータ品質モニタリング）** によって品質劣化を検知できる
- **Goal 0024（データインベントリ）** は Q1 着手を前提とする
- テーマ1〜3は相互に強化し合う（地方議会データの拡充が紐付けカバレッジの分母を押し広げる）

## リスクと対応

| リスク | 影響 | 対応 |
|-------|------|------|
| 地方議会スキーマで国会スキーマが不足 | テーマ3 遅延 | Goal 0012 でギャップを事前洗い出し |
| 衆議院賛否データの構造化が困難 | テーマ2 遅延 | LLM 抽出に切り替え・委員会議事録をソースに |
| Snowflake 同期コスト | テーマ4 コスト超過 | 差分同期・更新頻度を週次に抑える |
| 地方議員マスタ投入の人的コスト | テーマ3 遅延 | 選挙結果データからの自動投入を優先 |
| 紐付け精度改善の長期化 | テーマ1 継続課題 | フィードバックループで継続改善・MVP 到達で一旦打ち切り |
