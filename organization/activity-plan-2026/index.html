    直近1年の活動計画（2026年4月〜2027年3月）¶

    概要¶

    本ドキュメントは、政治活動の透明性向上を目的として運営している sagebase（ https://sage-base.com ）における 2026年度（2026年4月〜2027年3月）の活動計画をまとめたものです。これまで蓄積してきた国会データを基盤に、データの意味化・対象範囲の拡大・提供経路の多様化の3点を重点領域として位置づけ、市民・研究者・報道機関・行政機関のみなさまにとってより利用しやすいデータ基盤の整備を進めます。

    これまでの到達点（2026年3月時点）¶

    sagebase はこれまで国会（衆議院・参議院）を対象に、以下の取り組みを完了しています。

    • 国会会議録 API を通じた発言データの全量取得（約1,120万件）
    • 発言者と政治家情報の紐付け（国会全体）
    • Google BigQuery 上でのパブリックデータセット公開
    • データ加工プロセスにおけるレコード単位の出典情報の付与
    • データ品質の継続モニタリング体制の整備
    • 参議院における議案ごとの個人投票データの構造化
    • 利用者からの誤り・改善提案を受け付けるフィードバック窓口の設置

    これにより、「いつ・どこで・誰が・何を発言し、どのような賛否を示したか」を機械可読な形で辿れる基盤が整ってきました。一方で、国会に比べ地方議会のデータ整備は大きく遅れており、また利用経路も一部の技術者向けに限定されているなどの課題が残っています。2026年度はこれらの課題に対応するため、4つのテーマで取り組みを進めます。

    2026年度の4テーマ¶

    # テーマ 目指す状態
    1 発言者と政治家情報の紐付けカバレッジ向上 国会での紐付け精度をさらに高め、地方議会にも拡張できる基盤を整備
    2 議案に対する賛否データの全面整備 衆参両院および地方議会における賛否データの公開
    3 地方議会データの順次拡充（都道府県議会→市区町村議会） 47都道府県議会および政令指定都市議会のデータ整備
    4 データ提供方法の拡充 Snowflake、MCPサーバー、API など、多様な経路での利用環境整備

    テーマ1: 発言者と政治家情報の紐付けカバレッジ向上¶

    背景と課題¶

    国会の発言データに対する政治家情報の紐付けは一次整備を完了しましたが、外部利用に耐える品質まで引き上げるには以下の課題が残っています。

    • 紐付け状況の可視化不足: どの会議体・政治家・時期に紐付けが不足しているか、外部から確認できる指標がない
    • 地方議会への適用未検証: 国会向けに構築した紐付けロジックが地方議会議員にそのまま適用できるかは未確認
    • 地方議員情報の不足: 地方議員の基礎情報そのものが整備されていないため、発言と政治家を結びつけようがない
    • エッジケースの継続改善: 同姓同名、結婚・改名、所属変更など、単純な照合で解消できない事例への継続対応が必要

    方針¶

    1. カバレッジ指標の標準化と公開: 発言単位・政治家単位・会議体単位での紐付け率を定期的に算出し、外部から閲覧可能なダッシュボードで公開します
    2. 紐付け基盤の汎用化: 国会向けに特化しているロジックを、議会種別に依存しない形に整理し直します
    3. 地方議員情報の逐次整備: 地方議会データの拡充と連動する形で、議員基礎情報の整備を進めます
    4. フィードバックループの活用: 利用者からの誤り報告を紐付け精度の継続改善に組み込みます

    年度末目標¶

    • 国会発言における紐付けカバレッジ 99% 以上の達成
    • 地方議会（整備済み自治体）における紐付けカバレッジ 95% 以上の達成
    • 紐付けカバレッジダッシュボードの一般公開

    テーマ2: 議案に対する賛否データの作成¶

    背景と課題¶

    参議院における議案ごとの個人投票データは構造化を完了しましたが、国会全体および地方議会を対象とした場合、以下が未整備です。

    • 衆議院議案の賛否: 衆議院では個人投票の公開形式が構造化に適しておらず、追加的な取得・抽出処理が必要です
    • 地方議会の賛否: 一部の先行研究データ（東京都議会を対象とした学術データセット等）を除き、ほぼ整備されていません
    • 会派方針との差分分析: 会派の公式方針と個人投票との食い違い（いわゆる造反）を機械的に検出する仕組みが未実装です
    • 議案と投票レコードの照合精度: 議案名の表記ゆれなどにより、全量の結合ができていません

    方針¶

    1. 衆議院議案賛否の体系的取得: 本会議議事録・議院運営委員会議事録から賛否情報を抽出するパイプラインを構築します
    2. 議案と投票の照合精度向上: 議案名の正規化と言語モデルによるあいまい照合を組み合わせ、結合率を引き上げます
    3. 会派方針との差分分析の提供: 会派方針の推定と個人投票との差分を可視化するレイヤーを整備します
    4. 地方議会賛否データの取得: 公開されている学術データセットの取り込みに加え、公開情報を持つ自治体から順次収集を行います

    年度末目標¶

    • 衆議院における 2020 年以降の賛否データの完全取得
    • 参議院における議案と投票の結合率 95% 以上の達成
    • 会派方針との差分分析機能の提供開始（国会対象）
    • 地方議会賛否データの対象自治体 5 以上への拡大

    テーマ3: 地方議会データの順次拡充（都道府県議会→市区町村議会）¶

    背景と課題¶

    国会のデータ整備は進む一方、地方議会は以下の特性を持ち、一律の手法では整備できません。

    • 日本には 47 の都道府県議会、20 の政令指定都市議会、約 1,700 の市区町村議会が存在します
    • 議会活動情報の公開方法は自治体ごとに大きく異なり、統一的な API は存在しません
    • 共通の会議録システムを採用する自治体群が存在する一方、独自システムや PDF のみの配信、ウェブ非公開といった自治体も少なくありません

    このため、規模・共通システム採用状況・公開形態を考慮した段階的な展開が必要です。本年度は都道府県議会を先行して整備し、並行して政令指定都市議会への展開を進めます。

    方針¶

    1. 先行自治体での検証完了: 京都市議会を対象に、地方議会データを受け入れるためのデータ構造上の課題を洗い出し、必要な拡張を実施します
    2. 都道府県議会の順次展開: 共通の会議録システムを採用している自治体から優先的に、発言・会議体・議員情報を取得します
    3. 政令指定都市への展開: 京都市での知見をもとに、横浜市・大阪市・名古屋市など主要都市へ展開します
    4. 地方議員情報の整備: 議会公式サイトおよび選挙結果情報から、現職議員の基礎情報を整備します
    5. 取得困難事例の記録: 整備が進まなかった自治体について、その理由を記録し、今後の代替手段検討や政策提言の材料とします

    年度末目標¶

    • 京都市議会における発言・議案・議員情報の完全取得
    • 都道府県議会 47 のうち 30 自治体の発言データ整備
    • 政令指定都市 20 のうち 10 市の発言データ整備
    • 整備済み自治体における現職議員の基礎情報 100% 整備

    テーマ4: データ提供方法の拡充¶

    背景と課題¶

    現在、sagebase のデータは Google BigQuery のパブリックデータセットを主な提供経路としていますが、以下の課題があります。

    • Snowflake 利用者への提供がない: 企業や行政機関の一部では Snowflake が標準のデータ基盤であり、BigQuery 以外での提供形態が求められています
    • AI エージェントからの利用導線がない: 近年急速に広がりつつある生成 AI エージェントから sagebase データを直接参照する手段がありません
    • 軽量な API 提供の不在: BigQuery は大規模分析に適していますが、ウェブサイトやアプリケーションから単発で問い合わせる用途には重く、軽量な REST API の需要があります
    • データカタログの不足: どの領域にどれだけのデータが蓄積されているかを外部から俯瞰しづらく、利用促進の妨げになっています

    方針¶

    1. Snowflake Marketplace での公開: BigQuery との同期パイプラインを整備し、主要データセットを Snowflake Marketplace で提供します
    2. MCP サーバーの提供: Model Context Protocol に準拠したサーバーを公開し、生成 AI エージェントから sagebase データを直接参照できるようにします
    3. Popolo 準拠 API の提供: 政治情報の国際標準スキーマである Popolo に準拠した軽量 REST API を公開します
    4. データインベントリの可視化: データ蓄積状況を地図・グラフで俯瞰できるダッシュボードを一般公開します
    5. 利用事例の収集と共有: 各提供経路における利用事例を収集・共有し、次年度以降の優先順位付けに反映します

    年度末目標¶

    • Snowflake Marketplace での主要データセット公開
    • MCP サーバー v1.0 の提供（政治家検索、発言検索、議案検索を含む3機能以上）
    • Popolo 準拠 API のアルファ版公開
    • データインベントリダッシュボードの常時公開

    四半期別マイルストーン¶

    第1四半期（2026年4月〜6月）: 基盤整備¶

    テーマ マイルストーン
    1. 紐付けカバレッジ向上 カバレッジ指標の定義およびダッシュボード初版の公開
    2. 議案賛否データ 参議院における議案-投票結合率 95% の達成／衆議院賛否取得の設計確定
    3. 地方議会拡充 京都市議会におけるデータ構造上の課題解消
    4. 提供方法拡充 データインベントリダッシュボードの公開／MCPサーバーのアルファ版公開

    第2四半期（2026年7月〜9月）: 横展開の開始¶

    テーマ マイルストーン
    1. 紐付けカバレッジ向上 紐付け基盤の汎用化完了
    2. 議案賛否データ 衆議院賛否データの取得開始（2024年以降分）／会派方針差分分析のアルファ版公開
    3. 地方議会拡充 都道府県議会 10 自治体の発言データ整備完了
    4. 提供方法拡充 Snowflake Marketplace 公開準備完了／MCPサーバー v1.0 公開

    第3四半期（2026年10月〜12月）: カバレッジ拡大¶

    テーマ マイルストーン
    1. 紐付けカバレッジ向上 地方議会（整備済み自治体）における紐付けカバレッジ 90% 達成
    2. 議案賛否データ 衆議院における 2020 年以降の賛否整備完了／地方議会賛否データ 2 自治体整備
    3. 地方議会拡充 都道府県議会 25 自治体／政令指定都市 5 市の整備
    4. 提供方法拡充 Snowflake Marketplace での公開／Popolo 準拠 API の設計確定

    第4四半期（2027年1月〜3月）: 品質向上と公開¶

    テーマ マイルストーン
    1. 紐付けカバレッジ向上 国会 99%、地方議会（整備済み）95% の達成
    2. 議案賛否データ 会派方針差分分析 v1 公開／地方議会賛否データ 5 自治体整備
    3. 地方議会拡充 都道府県議会 30 自治体／政令指定都市 10 市の整備
    4. 提供方法拡充 Popolo 準拠 API のアルファ版公開／年度活動報告の発行

    リスクと対応方針¶

    リスク 想定される影響 対応方針
    地方議会データを受け入れるためのデータ構造の不足 テーマ3 の進捗遅延 京都市議会での検証段階で課題を事前に洗い出し
    衆議院における賛否データの構造化の困難さ テーマ2 の進捗遅延 言語モデルによる抽出への切り替え、委員会議事録を補助的情報源として活用
    Snowflake 上でのデータ同期にかかる費用増大 テーマ4 の費用超過 差分同期と週次更新によるコスト抑制
    地方議員情報の整備にかかる人的負荷 テーマ3 の進捗遅延 選挙結果情報からの自動投入を優先
    紐付け精度改善の継続的な負荷 テーマ1 の長期化 フィードバック経由の継続改善に切り替え、一定水準での整備完了を宣言

    年度末KPI¶

    指標 目標
    発言紐付けカバレッジ（国会） 99% 以上
    発言紐付けカバレッジ（地方議会・整備済み自治体） 95% 以上
    議案-投票照合率（国会） 95% 以上
    都道府県議会整備数 30 / 47 自治体
    政令指定都市議会整備数 10 / 20 市
    データ提供経路数 4 経路（BigQuery / Snowflake / MCP サーバー / Popolo 準拠 API）
    外部利用事例の蓄積 10 件以上

    おわりに¶

    本計画は、2026年度における sagebase の活動の指針を示すものですが、政治情報の透明化は本プロジェクトのみで完結する課題ではありません。市民のみなさま、研究機関、報道機関、そして議会事務局をはじめとする行政機関のみなさまからのフィードバック・ご協力があってはじめて、より正確で使いやすいデータ基盤を築くことができます。本計画に関するご意見・ご提案はいつでも歓迎いたします。
